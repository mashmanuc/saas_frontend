## 1. Призначення
Цей маніфест встановлює єдиний порядок роботи з текстами, перекладами та ключами i18n у M4SH Platform. Будь-який новий код, документація або тулінг мають відповідати даній політиці.

## 2. Базові принципи
1. **Джерело правди:** `src/i18n/locales/uk.json` визначає повну структуру ключів. Інші локалі повторюють структуру без відхилень.
2. **Без хардкоду:** користувацькі тексти зʼявляються тільки через i18n-ключі. ESLint-правило блокує прямі рядки.
3. **Домени та неймспейси:** ключі групуються за бізнес‑доменами (`calendar.*`, `auth.*`, `billing.*`, `notifications.*` тощо). Заборонено змішувати сутності в одному namespace.
4. **Snapshot-орієнтація:** переклади описують стани (екрани, повідомлення) та мають бути готові до повторного використання в різних UI.
5. **Масштабованість:** додавання нових мов не потребує рефактору існуючих контрактів.

## 3. Конвенції іменування
- `module.feature.state.field`: приклад `calendar.lessonModal.title`.
- Плейсхолдери називаємо зрозуміло: `{studentName}`, `{count}`.
- Для загальних компонентів використовуємо namespace `ui.*`.

## 4. Процес додавання/оновлення ключів
1. Dev додає структуру в `uk.json` (та, за можливості, переклад в `en.json`).
2. Запускає `pnpm i18n:check`.
3. Якщо ключ новий — створює таск у беклозі «Translation needed».
4. Після ревʼю ключ потрапляє у всі локалі (можна тимчасово залишити `TODO_translate`).
5. CI/cron слідкує за незакритими ключами.

## 5. Валідація та інструменти
- `scripts/i18n-check.mjs` (розширений):
  - перевіряє всі локалі зі списку `SUPPORTED_LOCALES`.
  - сигналізує дублікат ключа.
  - розділяє «відсутній ключ» і «порожнє значення».
- ESLint правило `no-hardcoded-translations`.
- Admin dashboard «Missing translations» (доступ ролям `i18n_manager`).

## 6. CI/CD політика
- Pre-commit хук: `pnpm i18n:check`.
- GitHub Actions job `i18n-check` обовʼязковий для PR.
- Cron job (щотижня) публікує звіт у `#l10n-status`.

## 7. Ролі та відповідальність
- **Language Platform Lead:** володіє маніфестом, контролює зміни.
- **Команди FE/BE:** зобовʼязані запускати перевірки, не мерджити з помилками.
- **Product Ops / Localization:** обробляють беклог перекладів із адмінського тулінгу.

## 8. Відхилення
Будь-яке відхилення (наприклад, тимчасовий текст без перекладу) документується в PR описі й має дедлайн для виправлення. Відхилення без погодження з Language Platform Lead заборонені.

## 9. Оновлення маніфесту
- Будь-які зміни обговорюються на архітектурній раді.
- Version bump у шапці + changelog у PR.
