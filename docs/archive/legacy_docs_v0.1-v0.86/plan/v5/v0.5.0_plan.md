10 наступних кроків
1. Привести моделі в порядок (базова доменна схема)

Мета: перед тим, як ліпити адмінку/чат/дошки, зафіксувати, що в нас буде в домені.

Перевірити й задокументувати моделі: User, TutorProfile, StudentProfile, Classroom, ClassroomMembership, TutorStudentRelation.

Додати/підчистити абстрактні базові класи для типових сутностей (наприклад, TimestampedModel, можливо BaseProfile, BaseContent).

Оновити коротку діаграму (навіть текстом у docs/domain_model.md).

2. Профілі користувачів (Tutor / Student profile v1)

Мета: зробити нормальні профілі, а не тільки email + ім’я.

Бекенд:

Перевірити й доробити TutorProfile, StudentProfile (subjects, опис, timezone, рівень учнів, чи показувати в пошуку).

Ендпоінти типу: GET/PUT /api/me/profile/ (одноразово повертає потрібні поля в залежності від ролі).

Фронт:

Простий ProfileView (наприклад /profile) з можливістю змінити основні поля.

3. Мінімальна адмінка для тебе (не для всіх)

Мета: зробити інструмент для тебе/адмінів, щоб не лазити в Django shell.

Бекенд:

Використати Django admin + (опційно) легенькі DRF-ендпоінти типу /api/admin/users/, /api/admin/classrooms/ (тільки для SUPERADMIN/ADMIN).

Фронт:

Простий AdminDashboardView:

список користувачів (email, роль, активність),

список класів (назва, тьютор, кількість учнів).

Це дасть можливість дивитися й вручну правити найважливіші сутності.

4. Абстракція для завдань / тестів (без справжнього UI)

Мета: закласти основу для майбутніх тестів, але не витрачати тиждень.

Бекенд:

Створити абстрактну модель BaseTask / TaskTemplate (тип, предмет, текст, метадані).

Модель Assignment (що дав тьютор кому і коли).

Модель Submission (відповідь студента).

Ніякого складного рендера, просто JSON-поля + мінімальні ендпоінти:

POST /api/tutor/assignments/

GET /api/student/assignments/

На фронті поки зробити заглушки: списки “Мої завдання”, без красивого контенту.

5. Віртуальні “дошки” (whiteboard) – модель + заглушка

Мета: закласти фундамент під дошки, але не робити Excalidraw v2.

Бекенд:

Модель WhiteboardSession:

id, title, owner (tutor), classroom/tutor_student_relation, data (JSON).

Ендпоінти:

GET /api/whiteboards/ (мої сесії),

POST /api/whiteboards/ (створити пусту),

GET/PUT /api/whiteboards/<id>/ (оновити JSON).

Фронт:

Простий екран: список дошок + кнопка “Створити”.

Усередині – поки що просто textarea з JSON / текстом (заглушка, пізніше підключимо нормальний компонент).

6. Чат: модель повідомлень + базовий API

Мета: зробити структуру чату, щоб потім можна було підключити реальний UI / WebSocket.

Бекенд:

Моделі:

Conversation (між тьютором і студентом або в межах класу).

Message (author, text, created_at, is_read).

Ендпоінти:

GET /api/conversations/ (мої),

GET /api/conversations/<id>/messages/,

POST /api/conversations/<id>/messages/.

Фронт:

Дуже простий “чат” у вигляді списку повідомлень + форма вводу.

Без realtime, просто fetch + post + refresh.

7. Перевірити й спростити permission-и / ролі

Мета: щоб не роз’їхалося, хто що може.

Пройтися по міксинах типу TutorOnlyMixin, StudentOnlyMixin, перевірити, що:

студент не бачить чужі classrooms;

тьютор не чіпає не своїх студентів;

адмін може все дивитись, але НЕ плутати це з тьюторським інтерфейсом.

Задокументувати в docs/permissions.md: хто що може робити в 1–2 таблицях.

8. Тестові сценарії (ручні чеклісти + перші автотести)

Мета: закрити базові флоу, щоб не боятися нових релізів.

Написати текстовий чекліст (md-файл) по основних потоках:

реєстрація студента / тьютора;

студент self-learning → знайти тьютора → надіслати запит → тьютор приймає;

перегляд класів тьютора;

створення assignment + перегляд студентом.

Бекенд: написати хоча б 3–5 pytest-тестів:

auth + reloadUser flow,

/api/tutors/public-list/,

/api/student/request_tutor/,

/api/tutor/accept_request/,

/api/tutor/classrooms/.

9. Нормальний “Profile / Settings” на фронті

Мета: щоб користувач міг сам себе налаштувати, а не через адміна.

В одному екрані:

базові дані (ім’я, timezone),

длЯ репетитора: предмети, рівні учнів, ціна/валюта, “показувати в пошуку”.

для студента: цілі навчання, клас/рівень.

Підключити це до ендпоінта з кроку 2.

10. Підсумковий “v0.5.0 Roadmap Review” + фіксація

Мета: не загубитися в історії версій.

Коли 1–9 будуть хоча б частково зроблені:

створити docs/v0.5.0_release_notes.md,

docs/v0.5.0_architecture.md (додати чат, дошки, tasks),

git tag v0.5.0.

Окремо зберегти список фіч, які ще “в майбутнє” (AI-пояснення задач, складні борди, відео-уроки і т.д.), щоб вони не плутались з базовим MVP.
